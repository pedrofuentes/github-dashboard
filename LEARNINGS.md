### [2026-06-29] Deck filtered-reorder gotcha: always pass the full repos list + repoFilter — never pre-filter before passing to the Deck
**Context**: PR #630 introduced Deck row drag-to-reorder. A 🔴 Sentinel finding caught that passing a pre-filtered repos array to the component broke the filter gate.
**Learning**: The Deck persists a **full-fleet** repo order. The component receives `repos` (the full list) and `repoFilter` (an optional `Set<string>`) and internally narrows what is *displayed*. If you pre-filter the list outside the component (e.g. `repos.filter(r => repoFilter.has(r.nameWithOwner))`) and pass the result as `repos`, the component never sees the full fleet. When it checks `filterActive = repoFilter !== undefined`, the flag fires correctly — but the order it would persist only covers the visible subset, silently corrupting the positions of the hidden repos. The full repo list must travel alongside the filter so the gate can compare "am I looking at the whole fleet?" correctly. The same principle applies to any component that owns a full-fleet persistence contract (cf. ADR-025's arrange-guard for Dashboard tiles, ADR-032 for Deck row order).
**Impact**: Never pre-filter a list before passing it to a component that persists full-fleet order — let the component do the narrowing internally so its filter gate has access to the complete set. When reviewing a component that both filters and persists order, verify the reorder handler receives the *unfiltered* source list.

### [2026-06-22] react-grid-layout fires `onLayoutChange` on ANY layouts change — even in display mode with no drag — so a filtered/partial layout must gate persistence
**Context**: Phase 3 Wave C — adding a repo-scope filter that projects the Dashboard grid down to a subset of tiles (`DashboardView`). The grid uses `compactType="vertical"`.
**Learning**: RGL's `onLayoutChange` is NOT drag-specific. It fires on mount, on responsive breakpoint switches, and whenever the `layouts` prop changes — including when a filter narrows the rendered tile set. With vertical compaction, the reported layout is the *compacted geometry of only the visible subset*, which collapses the gaps left by filtered-out (or hidden) tiles. If that payload is merged back and persisted, it silently saves a partial/compacted layout over the real full-layout geometry. The fix is to gate persistence whenever the rendered set is a strict subset: unwire the grid callback (`onLayoutChange={filterActive ? undefined : handleLayoutChange}`) AND early-return inside the handler (belt-and-suspenders), plus turn off `isDraggable`/`isResizable` and suppress the keyboard rail. Note `mergeLayoutGeometry` is full-layout-aware (a tile with no RGL item keeps its geometry), but that only protects tiles *absent* from the payload — it does NOT stop compaction from MOVING the visible ones, which is why the persist gate (not just the merge) is the real guard.
**Impact**: Any time the Dashboard grid renders a subset of the persisted tiles (repo filter, a future search, a hidden-tiles view), persistence MUST be gated — never trust an `onLayoutChange` emitted while the rendered set is partial. Test the guard by asserting storage is byte-unchanged after a simulated compaction emit, with a flushed positive control proving the same change DOES persist unfiltered.

### [2026-06-22] A "storage unchanged" assertion has no teeth unless you flush the 300 ms persist debounce first
**Context**: Phase 3 Wave C verification net (`DashboardView.arrange.test.tsx`), proving arranging is a no-op while filtered (stored layout byte-unchanged).
**Learning**: `useDashboardLayout` debounces the `saveDashboardLayout` write by 300 ms (`debounce(..., 300)`). A synchronous `localStorage.getItem` right after firing an arrange event sees the PRE-write baseline regardless of whether a write was scheduled — so a "storage unchanged" assertion passes with OR without the guard (the write simply hasn't flushed yet), giving a false green. To make the negative real, drive fake timers (`vi.useFakeTimers()` + `act(() => vi.advanceTimersByTime(400))`) — or unmount to trigger the flush effect — BEFORE reading storage, so a write WOULD have landed if the guard were absent. Pair the negative with a flushed positive control (same change, no filter) showing storage DID change, so the test proves the guard rather than the debounce latency.
**Impact**: Any test asserting a debounced side-effect did NOT happen must first advance past the debounce (or flush via unmount); otherwise it false-greens. This complements the existing learning to assert the persisted VALUE/boundary (never a synchronous spy count) — together they make both the positive and negative persistence assertions robust under Node 20's localStorage shim.

### [2026-06-20] Sentinel must run `prettier --check` on the CLEAN committed tree — a reformatted working tree can hide a formatting failure that CI then catches
**Context**: PR #181. The PR was reviewed by Sentinel and came back **APPROVED**, yet the merge-time CI `quality` gate failed on `prettier --check` (formatting). Root cause: during review the reviewer had run `prettier --write` (or `npm run format`) on the working tree before running the format check, so `prettier --check` passed against the **reformatted** tree — not against the tree that was actually committed and pushed. The committed tree still contained an unformatted file, which CI (running `--check` on a clean checkout) correctly rejected.
**Learning**: `prettier --check` only proves formatting if it runs on the **exact committed tree**. Running `prettier --write` (or `npm run format`) first mutates the working tree, so a subsequent `--check` validates the reformatted bytes, not what's in the commit — a false green. The lint/format step must be run on a **clean** tree (`git stash`/clean checkout, or simply never `--write` before `--check`), and any reformatting must be **committed** before the check is trusted. The merge-time CI gate (`--check` on a fresh checkout) is the catch-all backstop: a PR can be Sentinel-APPROVED yet still fail CI on formatting precisely because the reviewer checked a tree the commit doesn't contain. **APPROVED is necessary but not sufficient — the CI gate runs against the committed bytes.**
**Impact**: When validating formatting locally or in review, run `npm run lint` (which includes `prettier --check`) on the committed tree **without** a preceding `prettier --write`; if `format` changes anything, commit it and re-check. Don't treat a Sentinel APPROVE as proof the formatting gate will pass — confirm `prettier --check` was run on the clean committed tree (matching what CI checks out). This is the same class of "passes against the wrong tree/runtime" trap as the Node-version CI-only failures above.

### [2026-06-20] React unmount cleanup does not run on a hard page close — persistence must flush on beforeunload/visibilitychange
**Context**: Hardening the M10 dashboard-layout persistence (#142, alongside #124/#141). The debounced `saveDashboardLayout` write (300 ms) is flushed by a React `useEffect` cleanup on unmount, which covers in-app view switches.
**Learning**: A hard page close, reload, or navigation does **not** unmount the React tree, so the `useEffect` cleanup never fires and a layout change made within the debounce window is lost. Only DOM lifecycle listeners catch this teardown: `window` `beforeunload` and `document` `visibilitychange` (when `document.visibilityState === 'hidden'`, the reliable signal for tab discard/freeze on mobile and bfcache). Both must call the same `flush` wrapper so the pending write lands before the JS context is torn down. The `visible` transition is a no-op — only `hidden` tears the page down.
**Impact**: Any client-only persistence behind a debounce/throttle needs an explicit `beforeunload` + `visibilitychange→hidden` flush in addition to React unmount cleanup; relying on unmount alone silently drops the last change on a hard close. Test these branches by dispatching the DOM events (stub `document.visibilityState` via `vi.spyOn(document, 'visibilityState', 'get')`) and asserting the persisted value.

### [2026-06-20] The Vitest Web Storage shim must ALWAYS install a plain-object memory Storage — a feature-detect early-return keeps jsdom's spy-hostile exotic Storage
**Context**: Fixing #124 — `vi.spyOn(localStorage, 'setItem')` recorded **0 calls** after a successful write on CI's Node 20 (and, locally, for `sessionStorage` on Node 25). `src/test/setup.ts`'s `ensureWebStorage` early-returned whenever `setItem` was already a function, so the jsdom-provided `Storage` survived.
**Learning**: jsdom's `Storage` is an **exotic object** whose named-property setter intercepts `setItem`; spying it returns a spy that is never invoked (the real write still happens via the exotic setter), so spy-count assertions silently see 0. The feature-detect early-return is the trap: it "preserves a working implementation" that is in fact spy-hostile. Dropping the early-return and unconditionally installing the plain-object memory `Storage` (for both `localStorage` and `sessionStorage`) is behavior-preserving for get/set/remove/clear and makes `vi.spyOn(storage, 'setItem')` record calls normally. This is test-only infra.
**Impact**: Never feature-detect-and-skip the storage shim. Install the plain-object memory `Storage` unconditionally so storage spies work identically across Node 20 (jsdom shim path) and Node ≥21/25 (native `--localstorage-file`). Guard it with the `src/test/setup.test.ts` regression test (spy records a call after a write).

### [2026-06-20] Assert debounced side-effects by awaiting the flush and checking the persisted value/boundary — and reproduce CI-only failures on the pinned Node version
**Context**: PR #123 (M10 T3) opened green locally but its CI `quality` check failed: the #116 debounced-persist "coalesce" tests (`useDashboardLayout.test.ts`, `DashboardView.onLayoutChange.test.tsx`) asserted exactly one `localStorage` write, getting `expected [] to have a length of 1 but got 0`. Two earlier fixes failed to reproduce locally: (1) `await vi.advanceTimersByTimeAsync(300)` inside `act`, and (2) a "synchronous flush on `unmount()`" that asserted `vi.spyOn(localStorage, 'setItem').mock.calls`. Both passed on the local machine yet kept failing on CI. **Root cause**: CI runs **Node 20.18.x** while local dev was **Node 25**. Two Node-version-dependent effects combine only on CI: (a) Node 20 has no native `globalThis.localStorage`, so `src/test/setup.ts`'s memory-shim is used (Node 25 ships a native one, so this code path never runs locally); and (b) React 18's `unmount()` effect-cleanup (`useEffect(() => () => persist.flush(), [persist])`) runs `persist.flush()` in a microtask that settles **after** the synchronous assertion — so a spy read right after `unmount()` sees 0 calls. The product debounce/flush logic (300 ms, coalesce-to-one, flush-on-unmount) is correct; this was a test-timing/assertion defect.
**Learning**: Assert debounced/async side-effects by (1) **awaiting the flush deterministically** — `await act(async () => { unmount(); })` so effects + microtasks settle — and (2) asserting the **persisted result or the persistence boundary**, never a synchronous spy count. Read the actual stored value via `await waitFor(() => expect(JSON.parse(localStorage.getItem(KEY) ?? 'null')).toEqual(finalState))`, and for the "exactly one coalesced write" claim spy the boundary `saveDashboardLayout` (`vi.spyOn(await import('../lib/dashboard-layout'), 'saveDashboardLayout')`, set up **before** render so the hook's debounced reference resolves to the spy) and assert `toHaveBeenCalledTimes(1)` inside the `waitFor`. Keep the "no premature write" pre-assertion. Precise localStorage write semantics stay covered by the pure-util `src/lib/debounce.test.ts` and `src/lib/dashboard-layout.test.ts`. Critically: **reproduce CI-only failures by running the pinned Node version** (e.g. `/tmp/node-v20.18.3-darwin-arm64/bin` or via `nvm`/`fnm`) with the CI flags (`CI=true npx vitest run --pool=threads --poolOptions.threads.maxThreads=2 --poolOptions.threads.minThreads=2`); the same command passes under Node 25, which is exactly why three prior local validations were misleading.
**Impact**: When CI fails but local passes, first **match the runtime** (Node version) before re-litigating logic — runtime-conditional code paths (native vs. shimmed `localStorage`) and microtask-timing of effect cleanup are invisible on a newer local Node. Assert side-effects by awaiting (`await act(async)` / `waitFor`) and checking the persisted value/boundary, never a synchronous spy-count. "Passes locally" is necessary but NOT sufficient unless local matches the CI Node version.
